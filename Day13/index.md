## 有限状態マシン
### Q1 有限状態機械と電子計算機は何が同じで何が違いますか
A1 by gemini
有限状態機械（FSM: Finite State Machine）**と**電子計算機（コンピュータ）は、どちらも「入力を受け取り、内部の状態を変えながら、出力を出す仕組み」という根本的な仕組みが共通しています。

最大の決定的な違いは「外部メモリ（自由に出し入れできる記憶領域）を持っているかどうか」です。

---

## 共通点と相違点のまとめ

| 項目 | 有限状態機械 (FSM) | 電子計算機 (コンピュータ) |
| --- | --- | --- |
| **本質** | 内部状態の移動ルールをあらかじめ決めた計算モデル | メモリ上のプログラムを実行できる汎用計算機 |
| **記憶（メモリ）** | **内部状態のみ**（追加の作業用メモリがない） | **内部状態 ＋ 膨大な外部メモリ** (RAM / SSD) |
| **共通点** | ・入力、状態、出力の概念を持つ<br>

<br>・決定論的なルールに従って動作する<br>

<br>・現実の物理メモリは有限である | ・入力、状態、出力の概念を持つ<br>

<br>・決定論的なルールに従って動作する<br>

<br>・現実の物理メモリは有限である |
| **計算能力の限界** | 非常に限られる（例：カッコの対応ネストをカウントできない） | チューリング完全（理論上、計算可能な問題はすべて解ける） |
| **身近な例** | 自動販売機、信号機、エレベーター | PC、スマートフォン、サーバー |

---

## 主な違い：何が計算能力を分けるのか？

### 1. 「記憶の仕方」が違う

* **有限状態機械:** 「いまどの状態にいるか」という1つの状態しか保持できません。たとえば、「コインが100円入った状態」から「ボタンが押された状態」へ移るだけで、過去の操作履歴などを残す作業ノート（メモリ）を持っていません。
* **電子計算機:** CPU内部の状態（レジスタなど）だけでなく、**巨大なRAMやストレージ**を持っています。計算の途中でデータを読み書き・書き換えしながら複雑な処理を行えます。

### 2. 計算できる内容（表現力）が違う

* **有限状態機械:** メモリがないため、「過去に何回イベントが起きたか」を無制限に数えることができません。例えば、`(((...)))` のように任意に深くネストされた括弧のペアが正しいか判定することは、状態数が足りなくなるためFSMには不可能です。
* **電子計算機:** メモリを使ってカウントやスタック操作ができるため、複雑なプログラミング言語の解析や高度なアルゴリズムを実行できます。理論的には「チューリングマシン」と同等の能力（チューリング完全）を持っています。

---

> **視点を変えた興味深い事実**
> 理論上、現実の電子計算機も「メモリ容量が物理的に有限（例: RAM 16GBなど）」であるため、膨大ではあるものの**「きわめて巨大な有限状態機械」の一種**とみなすこともできます。
> しかし、実用上・情報科学上のモデルとしては、両者は明確に区別されています。

## 3つの計算モデルの能力比較

最も能力が低い（制限が厳しい）ものから、最も能力が高い（何でもできる）ものへと順に解説します。

### 1. 有限状態機械 (FSM / 有限オートマトン)

* **対応する言語（チョムスキー階層）:** 正規言語 (Type-3)
* **記憶の仕組み:** **なし（現在の状態のみ）**
* **特徴:**
過去の履歴を記録するメモ帳を持っていません。「今どの状態にいるか」だけを頼りに動きます。
* **できること:**
単純なパターンの検索。例えば「`0`と`1`が交互に来るか」「特定のキーワードが含まれているか」などの判定は得意です（プログラミングで使う「正規表現」はこれに該当します）。
* **できないこと:**
回数を数えること。例えば「`A`が連続して出た回数と、同じ回数だけ`B`が出るか（$A^n B^n$）」は、回数を記憶できないため計算できません。

### 2. プッシュダウン・オートマトン (PDA)

* **対応する言語（チョムスキー階層）:** 文脈自由言語 (Type-2)
* **記憶の仕組み:** **スタック（積み上げ式メモリ）**
* **特徴:**
FSMに「スタック（Last In, First Out）」と呼ばれる特殊なメモリを追加したものです。机の上に本を積み上げ、一番上の本だけを見たり、取り除いたりできるイメージです。
* **できること:**
FSMが苦手だった「対応関係のカウント」ができます。例えば、数式における「開いたカッコ `(` の数と、閉じるカッコ `)` の数が一致しているか」を、スタックに積んだり降ろしたりすることで判定できます。プログラミング言語の構文解析は主にこの能力を使っています。
* **できないこと:**
スタックの一番上しか見られないため、「`A`の数と、`B`の数と、`C`の数がすべて同じか（$A^n B^n C^n$）」といった、3つ以上の要素が複雑に絡むチェックはできません。

### 3. チューリングマシン

* **対応する言語（チョムスキー階層）:** 帰納的可用言語 (Type-0)
* **記憶の仕組み:** **無限のテープ（ランダムアクセス可能）**
* **特徴:**
マス目が書かれた無限に長いテープの上を、読み書きヘッドが「右へ左へ」と自由に移動しながらデータを記録・書き換えできるモデルです。現在の私たちが使っているコンピュータの理論上のベースです。
* **できること:**
**「計算可能な問題」はすべて解くことができます。** $A^n B^n C^n$ のような複雑なカウントも、テープを前後に行き来して印をつけながら確認すれば解けます。

---

## 階層のまとめ（マトリョーシカ構造）

これらの関係は、上位のものが下位のものの能力を完全に包み込む構造になっています。

| 階層 | 計算モデル | メモリの特徴 | 解ける問題の例 |
| --- | --- | --- | --- |
| **Type-3** | 有限オートマトン (FSM) | メモリなし | `ababab...` (パターンの繰り返し) |
| **Type-2** | プッシュダウン・オートマトン | スタック（一番上のみアクセス） | `((()))` (2つの対応関係) |
| **Type-1** | *(線形拘束オートマトン)* | *(有限のテープ)* | *(Type-0とほぼ同じだがメモリ量に制限)* |
| **Type-0** | **チューリングマシン** | **無限のテープ（自由自在）** | `a^n b^n c^n` (複雑な依存関係やあらゆる計算) |

> ※Type-1の「文脈依存言語（線形拘束オートマトン）」は、チューリングマシンのテープの長さを「入力文字の長さに比例した有限の長さ」に制限したもので、階層の間に入ります。

つまり、「どこまで自由に過去のデータを読み書きできるか」という**メモリの自由度**が、そのままマシンの「頭の良さ（計算能力）」に直結しているのです。

## 1. 有限状態機械（FSM）で「できない」こと

> **例：カッコのネストが正しく閉じられているかの判定（`(((...)))`）**

FSMはスタック等のメモリを持たないため、カッコが「何重に開かれたか」を無制限にカウントできません。これを正しく判定するにはメモリ（変数やスタック）が必要です。

以下のコードは、FSM（メモリなし）では不可能な処理を、メモリ（カウンター変数）を使って解決しています。

=== "C言語"

```c
#include <stdio.h>
#include <stdbool.h>

// FSM単体では実装不可。整数変数(メモリ)で開いた数を数える必要がある
bool is_valid_parentheses(const char *s) {
    int count = 0; // ← FSMには持てない「可変の記憶」
    for (int i = 0; s[i] != '\0'; i++) {
        if (s[i] == '(') count++;
        else if (s[i] == ')') count--;
        if (count < 0) return false; // 閉じカッコが先行したらエラー
    }
    return count == 0;
}

```

=== "Python"

```python
def is_valid_parentheses(s: str) -> bool:
    # カウンター(メモリ)を使ってネストの深さを保持する
    count = 0
    for char in s:
        if char == "(":
            count += 1
        elif char == ")":
            count -= 1
        if count < 0:
            return False
    return count == 0

```

=== "Rust"

```rust
fn is_valid_parentheses(s: &str) -> bool {
    let mut count: i32 = 0; // 開いた回数を記憶する状態変数
    for c in s.chars() {
        match c {
            '(' => count += 1,
            ')' => count -= 1,
            _ => {}
        }
        if count < 0 {
            return false;
        }
    }
    count == 0
}

```

## 2. プッシュダウン・オートマトン（PDA）で「できない」こと

> **例：3つの記号の長さが等しいかの判定（$a^n b^n c^n$）**

PDAは「スタック（LIFO: 後入れ先出し）」を1つしか持てません。`a` の数だけスタックに積み、`b` でスタックから取り出すと、`c` をチェックするときには**スタックが空になり、最初の `a` の数が分からなくなってしまいます**。

これを解くには、チューリングマシンのように**配列やメモリ全体を自由に行き来してカウントする処理**が必要になります。

=== "C言語"

```c
#include <stdio.h>
#include <stdbool.h>
#include <string.h>

// PDA(1つのスタック)では解けない。3つのグループを順に走査して比較する
bool check_an_bn_cn(const char *s) {
    int i = 0, n_a = 0, n_b = 0, n_c = 0;
    while (s[i] == 'a') { n_a++; i++; }
    while (s[i] == 'b') { n_b++; i++; }
    while (s[i] == 'c') { n_c++; i++; }
    
    // 文字列の末尾であり、かつ長さがすべて一致するか
    return s[i] == '\0' && n_a > 0 && n_a == n_b && n_b == n_c;
}

```

=== "Python"

```python
def check_an_bn_cn(s: str) -> bool:
    # PDA（1つのスタック）では解けないため、各文字数をカウントして比較する
    import re

    # 正規表現（これも内部的にはチューリング全能な計算）やカウントで解決
    n_a = len(s) - len(s.lstrip("a"))
    rest1 = s[n_a:]
    n_b = len(rest1) - len(rest1.lstrip("b"))
    rest2 = rest1[n_b:]
    n_c = len(rest2) - len(rest2.lstrip("c"))
    rest3 = rest2[n_c:]

    return len(rest3) == 0 and n_a > 0 and n_a == n_b == n_c

```

=== "Rust"

```rust
fn check_an_bn_cn(s: &str) -> bool {
    let chars: Vec<char> = s.chars().collect();
    let mut i = 0;

    let mut n_a = 0;
    while i < chars.len() && chars[i] == 'a' { n_a += 1; i += 1; }

    let mut n_b = 0;
    while i < chars.len() && chars[i] == 'b' { n_b += 1; i += 1; }

    let mut n_c = 0;
    while i < chars.len() && chars[i] == 'c' { n_c += 1; i += 1; }

    // 全ての文字を消費し、かつ個数が一致しているか
    i == chars.len() && n_a > 0 && n_a == n_b && n_b == n_c
}

```

## 3. チューリングマシンでも「絶対にできない」こと

> **例：任意のプログラムが停止するか判定する（停止性問題 / Halting Problem）**

「どのようなプログラムと入力が渡されても、それが**無限ループするか正常終了するかを完璧に判定する関数** `will_halt`」をつくることは、理論上**絶対に不可能**であるとアラン・チューリングが証明しました（背理法）。

もしそんな関数が存在すると仮定すると、自分自身を呼び出して矛盾を起こすプログラム（パラドックス）が作れてしまうためです。

=== "C言語"

```c
#include <stdio.h>
#include <stdbool.h>

// 【不可能な関数】あらゆる関数と入力を受け取って停止するか判定する関数
// （理論上、これを正しく実装することは不可能）
bool will_halt(void (*func)(void*), void* arg);

// 矛盾を発生させる悪魔の関数
void paradox_target(void* arg) {
    // もし will_halt が「停止する」と判定したら、あえて無限ループする！
    if (will_halt(paradox_target, NULL)) {
        while(1); // 無限ループ
    }
    // もし「停止しない」と判定したら、そのまま終了（停止）する！
}
// → will_halt(paradox_target, NULL) はどのような答えを出しても矛盾する。

```

=== "Python"

```python
from typing import Callable, Any


# 【不可能な関数】どのような関数 f と入力 x を渡しても停止するか判別する
def will_halt(func: Callable[[Any], Any], arg: Any) -> bool:
    pass  # 理論上、全パターンに対応する中身を書くことは不可能


# 矛盾を引き起こす関数
def paradox(func_input):
    if will_halt(func_input, func_input):
        while True:
            pass  # 「停止する」と予想されたら無限ループする
    else:
        return  # 「停止しない」と予想されたら即停止する


# paradox(paradox) を実行した瞬間、will_halt は正解を出せなくなる！

```

=== "Rust"

```rust
// 【不可能な関数】関数ポインタと引数を受け取り、停止するか判定する
fn will_halt<F>(func: F, arg: &str) -> bool 
where F: Fn(&str) {
    unimplemented!("どのような強力な計算機であっても実装不可能");
}

fn paradox() {
    let target = |code: &str| {
        // will_halt が「停止する(true)」と判定したら無限ループに入る
        if will_halt(|_| {}, code) {
            loop {} 
        }
        // 「停止しない(false)」と判定したらそのまま終了する
    };

    // 自己言及させるとロジックが崩壊する
}

```

## 利用者が増えすぎた（7つを超えた）ときの処方箋

もし検討を進める中で「どうしても利用者の種類が10個くらいになってしまう…」となった場合は、次の2つのアプローチで整理します。

### ① 「継承（共通化）」を使う

例えば、「視覚障害者」「車椅子ユーザー」「子供」を個別に分けるのではなく、**「一般利用客」をベース（親）として、「優先利用客」という子アクターを作る**イメージです。基本的な要求（移動する）は同じで、差分（音声ガイダンスやボタン高さなど）だけを追加指定します。

### ② システムの境界（コンテキスト）を分ける

「エレベーターを使う人」と「エレベーターの故障を分析・管理するメーカーの社員」は、同じシステムで扱うべきではありません。

* **エレベーター制御システム**（現場で動くシステム）
* **遠隔遠隔監視・遠隔保守システム**（メーカーのクラウドシステム）

このようにシステム自体を分割することで、1つのシステムあたりのアクター数を **3〜5個程度** に抑えることができます。

## まとめ

モデリング時に意識すべきルールは以下の通りです。

* **最大数の目安:** 1システムにつき **5〜7種類以内**（できれば 3〜5種類 が理想）。
* **分類の基準:** 人の「属性（立場）」ではなく、「**システムへの目的・権限（何がしたいか）**」で分類する。
* **溢れた場合:** 共通点のあるアクターを統合（抽象化）するか、システム自体を2つに分ける。

この「最大7つまで」という枠組み（人間の認知限界でもある「マジカルナンバー7」の法則）を頭に置いてモデリングを進めると、仕様の抜け漏れを防ぎつつ、すっきりとまとまりのある設計書が作成できます！

これらの利用者を想定した場合、システムには以下の3つの切り口での機能要求が必要になります。

* **言語選択ボタン・自動判定:**
デフォルト言語（例: 設置国の言語）に加え、ワンタッチでの「日・英・中」切替機能。
* **国ごとの文字・点字規格対応:**
ディスプレイ表示のフォント（中国語の簡体字/繁体字、タイ文字、アラビア/デーヴァナーガリー文字等への拡張性）および、手すり・ボタン上の**点字規格のプリセット切替**（国によって点字の規格が異なるため）。

### B. 行事・運用モード切替（イベント運用者 3.1 向け）

* **イベント特化モード:**
* **ラッシュ・混雑モード:** 車椅子優先運転、開保持時間の自動短縮、通過階の自動設定。
* **VIP・国際会議モード:** 案内表示を一時的に「指定言語優先」にし、特定階への直行運転を有効化。
* **季節・行事演出モード:** 液晶画面のグラフィックや音声案内メッセージの変更（例: 祝祭日案内など）。

## 3. まとめ：このモデル化による設計上のメリット

1. **ソフトウェアのシングルソース化（共通化）**
国ごとに別々のプログラムを作るのではなく、「出荷時パラメータ」を変えるだけで14ヶ国以上の仕様へ化けさせることができます。
2. **現場での設営コスト削減**
「出荷時調整（工場）」と「現地設置調整（施工時）」のアクターを分けることで、現地の作業員が複雑なプログラミングを行わず、国コードや施設モードを選ぶだけで安全に立ち上げられるようになります。
3. **多言語×アクセシビリティの相乗効果**
「英語話者」かつ「視覚障害者」といった、属性の組み合わせに対しても、ベースの言語設定とアクセシビリティ設定を独立して掛け合わせることができる柔軟なアーキテクチャが実現します。
